## Synthesis

### Agreement

- **Do not reopen the kernel design** (Option A, typed DI tokens, packs-own-app-services). All four independently reached this; Opus and Codex explicitly looked for a reason to reopen it and did not find one; the `EventLog` port's implementation quality was repeatedly cited as evidence the design is executing well.
- **The CI/guardrail enforcement gap is real, understated by the original framing, and the single highest-leverage near-term fix.** All four confirmed independently (and the Coordinator verified directly) that `guard.ts`'s two "implemented" checks are literal no-op stubs, there is zero CI, and the plan's own named "biggest risk" guardrail (a release-set matrix) doesn't exist in any form.
- **"CI first" vs. "vertical slice first" was a false dichotomy.** Task 1 of the plan's own 10-task sequence (the packed-tarball boot harness, never built) is simultaneously the cheapest CI gate and the minimal composition proof, and should be built immediately, followed directly by a deeper named-scenario slice (real HTTP route → `AgentExecutor` → stream → reconnect → cancel → restart/replay → one allowed + one denied tool call through `ToolExecutor`) — not as competing priorities but as one staged program.
- **EventLog-as-kernel-port (locked decision C1) is sound as implemented**, with one narrow, cheap, agreed-on documentation fix: label `@jini/sqlite` explicitly as a default/opt-in adapter rather than a mandate, and document that pairing it with a second unrelated domain database does not give atomic cross-store transactions.
- **Package sprawl (23 packages vs. the locked 14) is real and should be contained via a formal admission manifest (quarantine/incubator status + named-consumer requirement), not resolved by wholesale deletion or full retro-litigation.** All four converged on some form of this; none endorsed deleting the speculative packages outright.
- **The Express/Fastify branch should not merge as currently shaped.** All four agree on this specific point. They diverge on the longer-term question (see Divergence).
- **Do not fully freeze all parallel worktree porting.** All four explicitly rejected a blanket freeze as wasted opportunity cost, favoring a narrower, scoped pause (new package admission + edits to the shared-contract packages `protocol`/`core` specifically) while most leaf-package work continues.
- **Codex's three new findings (kernel-noun drift via `ArtifactStoreToken`, the `@jini/core/internal` handler-export leak, no resource-lifecycle model) are real and load-bearing.** Every participant who reacted to them (Gemini, Opus, Primary) endorsed them; none were rejected.
- **Progress should stop being measured as "lines ported."** Opus's reframe ("N named scenarios green through the real packed composition") was independently echoed by Codex ("zero packed production-shaped scenarios green" is the operative fact, not 73.5K lines) and adopted into Primary's and Codex's final recommendations.

### Divergence

- **Should `@jini/sqlite`/Fastify-style optionality be embraced or resisted as a pattern going forward?** Gemini wants Fastify rejected outright and would prefer fewer speculative multi-adapter surfaces in general. Primary, Opus, and Codex would keep Fastify's branch alive as an explicitly experimental, non-default option (not promoted, not deleted) until a named consumer requires it — closer to how `@jini/sqlite`'s "default, swappable adapter" pattern already works for EventLog. This is a real, not fully resolved, difference in risk tolerance for optionality-by-default vs. minimalism-by-default.
- **How aggressive should the pause be?** Codex's Option 1 ("Two-gate stabilization sprint") explicitly pauses *new package admission and broad backend-port merges* until both gates are green — a real, if scoped, slowdown. Primary's Stage 0/1/2 plan is more permissive, only serializing edits to `protocol`/`core` specifically and letting broad backend-port work continue in parallel with Stage 1. Opus's Option 1 sits between the two. This is a genuine, quantifiable difference in how much near-term velocity should be spent on stabilization before resuming breadth — not yet resolved to a single number of days/weeks.
- **Total vaporware vs. red-spec-but-unimplemented (OD-sync mechanism):** Opus's framing was corrected by Codex mid-round (a red spec test does exist), a minor but real factual delta that changes the size of finding F1's remaining lift.

### Unique Insights

- **Gemini's "framework vs. library" blind spot** — Jini's shape (a DI container that boots and calls your code) makes it an IoC framework, not a library OD calls; retrofitting OD (currently an app) into a framework's plugin is a materially bigger lift than extracting a callable library would have been. Nobody else raised this, and it was not resolved either way in Round 2 — flagged here as worth a dedicated follow-up, not folded into the ranked findings above.

Surfaced only in Round 2, too late for the other three to react to directly; flagged as the top item for the next debate round or an immediate direct verification pass, whichever comes first.

### Round 3 New Findings (Claude Fable 5 — not present in Rounds 1-2, independently verified against source)

- **F-A — The durable EventLog is secretly a ring buffer, and eviction is invisible to replay (the single most serious concrete bug found in this whole process).** `packages/sqlite/src/event-log.ts:25` — `maxEntriesPerRun` defaults to 2,000, FIFO-evicted. A substantive run (every token/tool-delta is a durable row) overflows this in minutes. Worse: a fresh-cursor replay after eviction has occurred returns `{kind: 'ok', ...}` with **no gap signal** — a reconnecting client silently receives a run missing its own `start` event and everything before the eviction point. This reproduces, one layer down, precisely the failure the original architecture's C1 correction was written to prevent. The Stage 1 named scenario as currently scoped (short fixture) would not catch this.
- **F-B — The locked canonical event envelope (correction C2) was never built.** Spec: `{runId, eventId, opaqueCursor, protocolVersion, ts, kind, payload, durability}`. Shipped (`packages/protocol/src/events.ts:9-14`): `{id, event, data}` — the raw SSE frame shape C2 explicitly named as the anti-pattern to avoid — while the docstring claims compliance. No `runId` on the envelope means a multiplexed consumer can't attribute events to a run without external bookkeeping.
- **F-C — `ToolExecutor` has zero production call sites.** It exists and has tests, but nothing in the real execution path (`agent-executor.ts`) calls it — the live path only observes subprocess `tool_use`, the same passive behavior OD has today. The authz gate is unused dead code, not merely bypassable (extends Codex's internal-export-leak finding).
- **F-D — Idempotent restart doesn't work, and fixing it requires a kernel port-surface change.** Dedup keys live in an in-memory index only; retrying `start()` with the same idempotency key after a crash creates a duplicate run. The `EventLog` port has no way to enumerate existing runs, so `RunLifecycle` cannot hydrate on restart even in principle. This means the fix touches `@jini/daemon`'s port files, not just lifecycle code — Primary's Stage 1 scope ("serialize `protocol`/`core` only") is too narrow; see amended Stage 1 below.
- **F-E — The locked sequencing (ship the headless engine before the frontend) was inverted in practice.** `@jini/ui` alone measures ~31K src lines (67K with tests) — larger than the entire built backend engine combined (~24K src) — while the 49K-line backend spine remains unbuilt. Also confirms (independently of Opus/Codex) that `AGENTS.md`'s package inventory is stale by three unlisted packages (`mcp`, `diagnostics`, `metatool`).
- **F-F — The codebase's own self-graded quality scorecards are unreliable.** The same file containing the broken-restart logic (F-D) carries an inline `@overallScore 100/100` comment, as do other files with confirmed defects. Recommend treating these scores as unverified until Stage 1's scenario actually passes.

## Final Recommendation

Adopt a **two-stage program**, synthesized from all five positions (closest to Opus's and Codex's solution slates, amended per Fable 5's Round 3 findings):

**Stage 0 — days, no freeze required, start immediately:**
