## Multi-agent method — worked cleanly this time

Ran 4 Programmer-persona agents concurrently (keystone, db-port-finish, cli/memory-delta, symlink-fixes), each in its own `git worktree` under `.claude/worktrees/`, each with a fresh `pnpm install` (not symlinked `node_modules` — a deliberate deviation from the prior session's note, chosen because this session's tasks touch *multiple interdependent* `@jini/*` packages, and a fresh install correctly wires intra-worktree workspace symlinks to that worktree's own sibling packages; a symlinked-in `node_modules` would resolve cross-package imports back to the *parent* repo's packages instead). Zero collisions confirmed twice: once by construction (each agent's target files never appear in another agent's diff — checked via `git diff <branch-point>..HEAD --stat` against the true fork point, not against a since-moved-forward `main`, which shows false-looking deletions), once by four clean sequential `git merge --no-ff` with no conflicts.

**New gotcha found:** after merging a branch that changes a package's `package.json` dependencies (the keystone added `@jini/core`/`@jini/daemon`/`@jini/sqlite`/`@jini/http`/`express` to `node-host`'s deps), the **main working copy's own `node_modules` does not update itself** — a plain `pnpm install` (even "already up to date" per the lockfile) is still required before that package will build in the main tree. Caught this via a post-merge sanity build; not caught by any individual worktree's verification since each worktree had already run its own fresh install.

**Coverage command gotcha, reconfirmed and extended:** `--coverage.include='src/**'` alone does **not** enable coverage collection in this vitest version — it silently runs tests with no coverage report unless paired with a bare `--coverage` flag too (`vitest run --coverage --coverage.include='src/**'`). Caught this mid-session verifying the cli/memory branch (numbers looked plausible from test-pass-count alone, but no coverage table printed at all until the flag was added).

## How to port a barrel (the proven pattern, updated)

- Reference implementations: **`packages/sqlite/src/db/`** (data-layer port) and now **`packages/node-host/src/create-local-node-daemon.ts`** + **`packages/http/src/api-security-middleware.ts`** (assembly/composition-layer port — different shape: less "lift a decomposed module," more "identify the generic skeleton buried in a god-function and rebuild it against already-ported primitives"). Both: faithful lift where possible, de-brand, drop OD-product pieces, 100% coverage, `source-map.md` documents the drop-list and every design decision.
- Neutrality (hard, guard-enforced): NO `Open Design` / `OD_` / `open-design` / `/tmp/open-design` strings in `@jini/**`. `grep -rInE 'Open Design|OD_|open-design'` must be empty. **Note: `pnpm guard` is currently a no-op skeleton** (`scripts/guard.ts` — TODO comment, no real neutrality scan implemented yet) — the manual grep is the actual enforcement today, don't over-credit a green `pnpm guard` run.
- Coverage discipline: 100% on all four metrics, `--coverage` flag required (not just `--coverage.include`, see gotcha above). Command: `pnpm --filter <pkg> exec vitest run --coverage --coverage.include='src/**'`.
- Before porting a "god function," check what's already been independently ported as reusable primitives underneath it — the keystone task turned out to need far less new porting than expected because `@jini/core`'s DI kernel (`createDaemon`/`definePack`/`bindings`) and half of `@jini/http`'s security-predicate logic (`api-token-auth.ts`, `origin-validation.ts`) already existed from *other* tasks. Cross-check with `codebase-memory-mcp`'s indexed call-graph of OD (already indexed, 277k nodes, `Users-la-Desktop-Programming-OSS-Repos-open-design` project) to confirm a god-function's callees really are all product routes before assuming a large port is needed.
