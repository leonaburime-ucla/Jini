# Proposal: automation.ts / routine.ts — what's actually blocking the port, and what's a real product-policy question

**Status:** Proposal only — neither route pack was ported this round. This document corrects and sharpens `packages/http/source-map.md`'s pre-existing classification of both files, based on a direct, full read of the actual OD source (`apps/daemon/src/routes/automation.ts`, `apps/daemon/src/routes/routine.ts`, `apps/daemon/src/routines.ts`, `apps/daemon/src/automation-proposals.ts`, `apps/daemon/src/automation-templates.ts`, `apps/daemon/src/automation-ingestions.ts`, all on `leonaburime-ucla/open-design`'s `refactor/web-memory-slice` branch) — not a re-statement of the earlier recon.

## Finding 1: `automation.ts`'s real blocker is bigger than "needs OD's memory store," but it's the *same shape* of gap `@jini/memory` already solved once

The pre-existing table said automation.ts's route file is clean but "its two real dependencies, `automation-proposals.ts` and `automation-ingestions.ts`, are not self-contained: `automation-proposals.ts` imports `deleteMemoryEntry`/`readMemoryEntry`/`upsertMemoryEntry` from OD's `memory.ts` store." Confirmed exactly, plus more:

- `automation-proposals.ts` (344 lines) also imports `upsertUserAutomationTemplate` from `automation-templates.ts`.
- `automation-ingestions.ts` (545 lines) imports `createAutomationProposal` (back into `automation-proposals.ts`) and `getAnyAutomationTemplate` (from `automation-templates.ts`).
- `routine.ts` (see Finding 2) *also* registers `GET /api/automation-templates` and `GET /api/automation-templates/:id`, directly depending on `automation-templates.ts` — a dependency the original table's `automation.ts` row didn't surface because those two routes live in a different file.

So there are really three interlocking modules behind these routes, not two: `automation-proposals.ts` (344 ln), `automation-templates.ts` (336 ln), `automation-ingestions.ts` (545 ln) — 1,225 lines of non-route logic, none of it ported anywhere in this repo today.

**The good news:** the `deleteMemoryEntry`/`readMemoryEntry`/`upsertMemoryEntry` calls are structurally the *exact same three operations* `packages/http/src/memory.ts` (this same task's memory-routes port) already generalized as `MemoryNoteStore.deleteEntry`/`readEntry`/`upsertEntry`. A future `automation-proposals.ts` port would satisfy that dependency the identical way `memory.ts` did — a structural collaborator interface, not a direct `@jini/memory` import (still blocked by the same `UNLOCKED.md`/R7 constraint documented in `memory.ts`'s module doc). This is not a new kind of gap; it is the same gap, already solved once, just not yet applied here.

**The real, undecided part:** `automation-templates.ts`'s `BUILT_IN_AUTOMATION_TEMPLATES` (its first ~145 lines) is real OD product content — actual authored automation-prompt templates, the same category `host-tools.ts`'s `CATALOGUE` is *not* (a catalogue of tool names/probe logic is a generic mechanism; a curated list of "here are good automation prompts to run on a schedule" is editorial content). Does a ported `automation-templates.ts` ship with zero built-in templates (host supplies its own), or does the kernel need an extension point for a host to register default templates without editing kernel source? That is a real product-policy question this task does not have the authority to answer unilaterally — the same category of question `packages/memory/source-map.md`'s own "prompt-composition... explicitly left un-ported" note already drew a line around for a sibling concern.

**Recommendation:** port `automation-proposals.ts`/`automation-templates.ts` (built-ins stripped, structural memory typing) and `automation-ingestions.ts` as a dedicated follow-up task the same size and shape as `@jini/memory`'s own original port — not attempted here because it is net-new backend porting work (not a route-pack wrapping job), and doing it hastily inside this task would risk exactly the "forced, undignified port" the task brief warned against.

## Finding 2: `routine.ts`'s CRUD is blocked on a real persistence-port design decision, but its scheduler is a clean, already-portable, unrelated win

`routine.ts` (348 lines) does two things that need separating:

1. **Routine CRUD + run history**, backed directly by OD's raw `db.js` SQL functions (`getRoutine`/`insertRoutine`/`updateRoutine`/`deleteRoutine`/`listRoutines`/`listRoutineRuns`/`getLatestRoutineRun`/`getRoutineRun`) against `routines`/`routine_runs` tables. `packages/sqlite/source-map.md` already documents these tables as explicitly out of scope for the db-barrel port ("`routines`(+`routine_runs`+`routine_schedule_claims`)... flagged explicitly out of scope in this task's brief"). There is no persistence port for a routine anywhere in this repo today — porting the routes without one would mean either inventing a `RoutineStore` port unilaterally (a real design decision: what does "opaque scope" look like for a routine the way `RunLifecycle.contextRef` replaced OD's `{conversationId?, projectId?}`?) or hand-waving a fake in-memory store that silently drops OD's actual persistence guarantees (race-safe scheduled-slot claiming — see below). Also confirmed directly: `target.mode === 'reuse'` validates `body.target.projectId` via `getProject` — the same shape `active-context.ts`'s `resolveResource` DI pattern already generalizes, and the *smaller* of the two couplings (the DB access is the real blocker, not this one field).
2. **A `RoutineService`** (`routineService.nextRunAt`/`rescheduleOne`/`runNow`/`unschedule`), imported from `routines.ts`.

**`routines.ts` (726 lines) is a genuinely clean, already-portable generic scheduling engine — a real, positive finding.** Its only import is `node:crypto`'s `randomUUID`. It has no `@open-design/contracts` dependency (its `Routine`/`RoutineSchedule`/etc. types are explicitly a "local mirror... kept here so this service typechecks under NodeNext," per its own header comment — not a product coupling). Its `RoutinePersistence` interface (`list()`, `insertRun(run, options?)`, `updateRun(id, patch)`, `getLatestRun(routineId)`) is *already* an injected port, not a direct DB dependency — the same "host supplies the real collaborator" shape this whole task's other ports use. Its `RoutineRunHandler` callback is *already* how OD itself decouples "what does firing a routine actually do" (project/conversation creation, `startChatRun` dispatch) from the scheduler — the engine has no opinion on any of that. It implements real, non-trivial, DST-safe timezone math (`Intl.DateTimeFormat`-based wall-clock computation across hourly/daily/weekdays/weekly schedule kinds) and a documented race-safe "scheduled slot" persistence contract (a `ScheduledRunPersistenceError` distinguishing "a sibling daemon already won this slot" from a real failure) that is genuinely hard to get right and not something to casually reinvent.

**This engine does not belong in `@jini/http`** — it has no Express/route surface at all, the same way `@jini/daemon`'s `RunLifecycle` isn't in `@jini/http` either. It is a strong, well-scoped candidate for a future `@jini/daemon` (or a new, small dedicated package) port: "generalize `routines.ts`'s `RoutinePersistence`/`RoutineRunHandler`-driven scheduler, matching `RunLifecycle`'s existing precedent for a kernel-owned, storage-injected service." Not attempted here because (a) it is out of an HTTP-route-pack task's actual scope — porting a non-HTTP kernel service is a different package's deliverable — and (b) `routine.ts`'s own HTTP routes still couldn't be finished even with the scheduler ported, since the CRUD/persistence-port design question above is independent and unresolved.

**A confirmed, real bug, not carried-forward-worthy:** `GET /api/routines/:id`, `DELETE /api/routines/:id`, and `GET /api/routines/:id/runs` all lack the try/catch every sibling handler in the same file has (verified directly, not just re-citing the prior table's claim) — `GET`/`DELETE`/`GET .../runs` at lines 236, 268, and 292 respectively of `routine.ts`. A future port should fix this per this repo's established "fix a proven bug during a port" discipline, not reproduce it.

## What would need to happen before either ports

1. **A scoping decision on `automation-templates.ts`'s built-in content** (Finding 1) — ship zero built-ins with a host-extension point, or some other answer; not this task's call.
2. **A `RoutineStore`/persistence-port design** for `routine.ts`'s CRUD (Finding 2) — the same category of design work `@jini/daemon`'s `RunLifecycle` or `@jini/http`'s new `workspace-root.ts` required, sized for a dedicated task.
3. **Independently, and with no dependency on either of the above**, `routines.ts`'s scheduler engine is ready to port today as its own small, well-scoped task into `@jini/daemon` — flagged here as a genuine, low-controversy recommendation for whoever picks up automation/routines next.

## What this document is not

Not a claim that automation/routines has no generic core — the opposite: `routines.ts` is one of the cleanest, most obviously-portable pieces of logic found across this entire route-pack audit. The blocker is that the *route layer* around it (both files) needs real design decisions this task does not have the standing to make unilaterally, and the *other* backend modules it also depends on (`automation-proposals.ts`/`automation-templates.ts`/`automation-ingestions.ts`) are a same-sized porting task of their own, not a route-wrapping exercise.
