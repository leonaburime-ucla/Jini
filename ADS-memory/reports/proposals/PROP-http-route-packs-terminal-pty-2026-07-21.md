# Proposal: should `@jini/http` expose interactive terminal/PTY sessions, and through what authorization shape?

**Status:** Proposal only — no terminal/PTY route was ported this round. `packages/http/source-map.md`'s 2026-07-21 section on this task records the same verdict; this document is the reasoning behind it.

**Referenced from:** `packages/http/source-map.md`'s "2026-07-21 — route-pack audit" section, and the task brief that asked for a `chat/model proxy, artifact delivery, terminal/PTY, memory routes, automation/routines, DB operations, editor/host tools` audit.

## The problem

OD's `apps/daemon/src/routes/terminal.ts` (109 lines) mounts `GET /api/projects/:id/terminals`, `POST /api/projects/:id/terminals`, `GET /api/projects/:id/terminals/:tid/stream` (SSE), `POST .../stdin`, `POST .../resize`, `POST|DELETE .../kill`, all thin delegations to `apps/daemon/src/terminals.ts`'s `createTerminalService` (415 lines) — an in-memory session manager that spawns a real interactive shell via `node-pty` (`pty.spawn(shell, [], {cwd, env: {...process.env}, ...})`, defaulting to `$SHELL`/`/bin/bash` on posix or `ComSpec`/`powershell.exe` on win32) and fans its output out over SSE with a bounded reattach ring-buffer.

`packages/http/source-map.md`'s pre-existing routes-classification table already called this file row **#11, MIXED**: "Generic PTY/SSE session transport (list/create/stream/stdin/resize/kill), wrapped in `ctx.projectStore.getProject`/`ctx.projectFiles.resolveProjectDir` purely to resolve a spawn cwd. Would port cleanly once a generic 'workspace root resolver' port exists. Requires SSE (deferred). No lock between a `kill` and a concurrent `stdin`/`resize` on the same session id."

**Both named blockers are now resolved** — this same task built `packages/http/src/sse.ts` (a generalized version of `runs.ts`'s SSE machinery) and `packages/http/src/workspace-root.ts` (the generic workspace-root port, now consumed by `host-tools.ts`'s `POST /api/resources/:resourceRef/open-in`). Mechanically, `terminal.ts`'s routes could be ported today the same way `host-tools.ts`'s open-in route was. That is exactly why this is a proposal, not a straightforward defer-for-time note: the *stated* blockers are gone, and the remaining reasons not to port are real design/security decisions, not missing plumbing.

## Why this isn't just "port it, gate it through ToolExecutor, done"

The task brief's binding precedent is that any route pack with real command-execution stakes "MUST go through `ToolExecutor`'s existing deny-by-default authorization gate, not a new parallel path." `ToolExecutor.execute(principal, run, toolId, input, signal)` (`packages/daemon/src/tool-executor.ts`) is shaped for **one bounded call**: a policy authorizes it, an optional confirmation gates it, a handler runs once, and the result is either data or an error. That shape fits `db-ops.ts`'s `inspect`/`verify`/`vacuum` (this same task's DB-operations port) and `@jini/deploy`'s `deploy.publish` cleanly.

An interactive terminal does not have that shape. Its dangerous operation is not "the create call" — it's the *entire session's ongoing existence*: once a shell is spawned, every subsequent `POST .../stdin` write is arbitrary, unaudited, unbounded input to a real shell, for as long as the session lives (up to `terminals.ts`'s 30-minute idle TTL). Two designs are both technically "gated by ToolExecutor," but they mean very different things:

1. **Gate only `terminal.create` as a tool call**, then let `stdin`/`resize`/`kill` flow through ordinary same-origin-gated routes once a session id exists. This is what a naive, minimum-effort port would produce. It satisfies "the gate is in the call path" for creation, but the actual capability a bad actor wants — running arbitrary commands — is reachable through plain `stdin` writes that never touch `ToolExecutor` again. This is a materially weaker boundary than `db-ops.ts`'s or `deploy.publish`'s "every dangerous call goes through the gate," even though it can be described the same way in a changelog.
2. **Treat every `stdin` write as its own tool call.** This is closer to the real intent of "gate the dangerous thing," but is impractical: a terminal's whole value is low-latency interactive keystrokes; wrapping each one in `ToolExecutor.execute`'s async authorize/confirm/audit machinery would make it unusable, and `ToolDescriptor.requiresConfirmation` (a per-call "are you sure" prompt) makes no sense fired once per keystroke.

Neither option is obviously correct, and picking one silently — inside a route-pack porting task — would smuggle a real security-architecture decision into the kernel disguised as "this pack is done." That is exactly the outcome the task brief said not to force.

## Secondary, smaller-but-real blockers

- **`node-pty` is a new native compiled dependency.** Nothing in the current Jini workspace depends on a native addon requiring per-platform prebuilds. `packages/http/source-map.md`'s `host-tools.ts` section already flags this exact category of risk for a much smaller surface ("this repo's CI has no macOS/Windows runner and no installed copies of any catalogue entry") — a real PTY spawn is a strictly bigger version of that same gap, since there is no probe/fallback path the way `host-tools.ts` has; either `node-pty` loads or the daemon's terminal feature is entirely broken on that platform/CI runner.
- **A known, unfixed race in the origin.** `kill` has no lock against a concurrent `stdin`/`resize` on the same session id (source-map's own note). A real port should fix this rather than carry it forward, but that is more work layered on top of the authorization-shape decision above, not a reason to defer on its own.
- **Idle-exit and resource-lifetime semantics** (30-minute TTL, 512KB/64KB ring-buffer caps, exit-tail retention) are all sound as ported, but they are OD's chosen numbers, not validated defaults for a generic kernel — a host embedding this in a very different context (e.g. a CI-driven headless daemon vs. an interactive desktop app) may want different bounds, and the port would need to decide whether these become configurable or stay fixed.

## What would need to happen before this ports

1. **A Software-Architect-level decision on the authorization shape** for a *session-scoped* dangerous capability, not just a call-scoped one. Candidate shapes worth evaluating: (a) `ToolExecutor.execute('terminal.create', ...)` returns a session token that a lighter-weight, still-explicit "session capability" check (not full `ToolExecutor`) validates on every subsequent request against that session, closer to a scoped credential than a repeated tool call; (b) extend `ToolExecutor`/`ToolPolicy` with a first-class "session" concept so `stdin`/`resize`/`kill` still funnel through the same audit trail without per-keystroke confirmation overhead; (c) explicitly decide the kernel does **not** offer raw interactive shells at all, and any host wanting one builds it itself outside `@jini/http`, the same way `@jini/http`'s own `compat.ts` design decision explains why some OD surfaces are deliberately left for a future host to build fresh.
2. **A decision on `node-pty` as a workspace dependency** — including whether it belongs in `@jini/http` at all (an HTTP/SSE transport package taking on a native process-spawning dependency is itself worth a second look; `@jini/daemon` or a dedicated `@jini/terminal` package are both plausible alternate homes).
3. Whichever shape is chosen, the kill/stdin/resize race noted above should be fixed as part of the port (this repo's established "fix a proven bug during a port, don't carry it forward" discipline — see `packages/http/src/local-daemon-request.ts`'s dead-branch removal and `host-tools.ts`'s discriminated-union fix for precedent).

## What this document is not

Not a claim that terminal/PTY access is unsafe to ever offer, nor a recommendation to abandon it. The generic transport primitives it needs (SSE, workspace-root resolution) are now built and already proven by `host-tools.ts`'s port in this same task. The remaining gap is a real authorization-model design question this task's scope does not include the authority to resolve unilaterally.
