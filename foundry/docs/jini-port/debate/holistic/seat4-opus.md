# Seat 4 — Opus 4.8 — Whole-System Improvement (frozen before reading other seats)

## 1. End-to-end seams that don't fit (the important part)

**Seam A — three "agent/tool/run/task" vocabularies will collide and cause accidental coupling.**
- `@jini/agent-runtime` runs the *end-user's* coding CLIs: `Run` = an agent turn, `Agent` = a coding CLI, `Tool` = something the model calls mid-run.
- `project-runner` runs the *dev* agents building Jini: `Task` = an extraction unit, `Attempt`/`lease` = an execution, `Agent` = Codex/Claude doing dev work.
- `AI-Dev-Shop` defines a *delivery pipeline*: `Agent` = a role (Architect/Programmer), `Skill`/`Tool` = a capability, `Stage` = a step.
- The kernel `ToolRegistry` = end-user tools (`deploy.publish`).
Four domains, same four words. **Fix:** hard vocabulary firewall. Engine `@jini/*` owns {Run, Agent(=coding CLI), Tool(=end-user callable)}. Automation owns {Task, Pipeline, Stage, DevAgent, Skill}. A committed glossary + a lint: automation packages must never import `@jini/*` *domain* types for their task model, and the engine must never import automation types. **BUT one deliberate, documented reuse:** project-runner MAY depend on `@jini/agent-runtime` purely as the *spawn-and-stream-a-CLI mechanism* (so dev agents and end-user agents share one process/stream primitive) — while keeping its own Task/lease/ledger domain entirely separate. Share the mechanism, never the domain model. If this isn't stated now, someone will "reuse" the engine run-ledger as the dev-task ledger and weld the two together permanently.

**Seam B — the frontend and the kernel will grow two event schemas that drift.**
The kernel emits `EventLog` events; `@jini/chat-react` consumes a transport that yields events and folds them into messages. If chat-react defines its *own* event union (lifted from OD's `PersistedAgentEvent`) while the kernel/protocol defines another, you have two schemas to keep in sync forever — the classic drift seam, and OD already lives it (`PersistedAgentEvent` vs the daemon's run events). **Fix:** `@jini/protocol` is the *single* canonical run-event type. The kernel's `EventLog`, the `ChatTransport` port, and the AG-UI adapter are all **projections of the same protocol events** — chat-react consumes `protocol.RunEvent`, it does NOT invent a union. The transport port must carry the kernel's **replay cursor + cancellation** as first-class (the frontend's reconnect/resume is literally the kernel's replay contract). One event contract, three consumers, zero drift. Beware a *fourth* representation sneaking in via AG-UI: protocol is canonical, AG-UI is an optional projection.

**Seam C — "consume packed tarballs" fights the React dev loop.**
Decision 2 says consumers consume packed `@jini/*`. But OD *transpiles* `@open-design/components` from **source** during dev (verified in AGENTS.md) for HMR + CSS Modules. A consumer importing `@jini/chat-react` as a *built* tarball loses edit→HMR and inherits whatever CSS build assumptions the package baked in. **Fix:** (a) engine's own dev (reference-web) uses workspace source; (b) external consumers use published packages that ship proper ESM + type decls + source maps + **precompiled CSS themeable via CSS variables** (never require the consumer's bundler to process CSS Modules); (c) support `pnpm link`/workspace-override for a consumer actively co-developing a package. This is a React-package-specific packaging contract that the "packed tarball" rule glosses over.

**Seam D — the extraction tasks and the project-runner ledger are the same thing, with a chicken-and-egg.**
The plan's "first 10 tasks" and project-runner's ledger are one model, not two — the extraction tasks *are* ledger entries, and the plan's N/O gates *are* their validation commands. But project-runner is simultaneously (i) the tool that drives the extraction and (ii) a thing being built. **Fix:** bootstrap a *deliberately minimal* runner as Task 0 — a committed `tasks.json` + the N/O CI gates as plain scripts + manual claim — and only grow it toward leases/queues/sandboxing if real parallelism demands. AI-Dev-Shop already supplies the pipeline/governance; the missing piece is a *thin* execution/ledger loop, not the fancy git-CAS distributed runner. Don't build the control plane's cathedral before extracting the first package.

## 2. Frontend reuse model

Headless-hooks + slots + Vite is right, **but the biggest system-level correction is sequencing: the engine does not depend on the frontend packages, so decouple the tracks.** A consumer can embed the daemon + CLI + protocol and build its own UI — Tovu-Runner, Zana, and bolt.diy all do exactly this. "Agent-ready" means agents drive the engine over CLI/HTTP/MCP, not via React. Therefore `@jini/chat-react`/`renderers-react`/`components` are a **separable, lower-priority second track**; ship the headless engine first. This de-risks the timeline enormously, because the frontend work is huge and mostly independent: production-Vite is unstarted, and the chat god-components (ChatComposer 5.6k / ChatPane 4.3k) need real decomposition. Don't let a "done" kernel be blocked behind a nowhere-frontend. Concrete: define `ChatTransport` = protocol events (Seam B), keep the shell per-product, and treat chat-react as a convenience consumers may adopt or skip.

## 3. Automation layer

Separate repo is right; the three-way split is sound *if* the vocabulary firewall (Seam A) holds. Two improvements: (1) **project-runner reuses `@jini/agent-runtime` as its spawn primitive** (don't build a second CLI-spawning stack for dev agents). (2) **ADS-memory's `specs_as_built/` is a placeholder today** — the real, valuable version is a *committed, commit-stamped* architecture/dependency-graph export produced by graphify/cbm; wire a "context hydrator" so the dev agents refactoring Jini read a *fresh* dependency graph rather than hallucinating obsolete APIs. Missing piece: the runner has no story for "a dev-agent's change to `@jini/protocol` must re-run every consumer's compat test" — the compat matrix (Seam/cross-cutting below) is the automation's real job.

## 4. Desktop/sidecar + Tauri

`desktop-host` defers cleanly *because* the frontend track is decoupled (§2) — you don't need it until you ship a desktop product. The `RenderService` port is the right call (keeps PDF/deck export off Electron-lock; headless-Chromium impl for CI/Tauri). Tauri is realistically scoped as an experiment *only if* the `desktop-host` contract is defined when the first Electron host is built (define the interface with one impl, add Tauri as the second) — don't define a host contract from a single impl (that's the "one consumer = OD-shaped" trap applied to hosts). Keep it low-priority as decided.

## 5. Cross-cutting

**Release/versioning is the sleeper risk.** 14 independently-versioned packages × N external consumers = a compat matrix nobody can hold. **Fix: lockstep-version the whole `@jini/*` set** (one version, released as a train), so a consumer pins a single `@jini` version and internal cross-package compat is guaranteed by construction. Reserve semver *meaning* for `@jini/protocol` specifically (the wire/consumer contract). Provide a local registry (verdaccio) or `pnpm pack` loop so consumers test pre-release. **Observability:** a `TelemetrySink` provider fed by the kernel EventLog (runs/cost/latency/failures) + the API-snapshot/patch-canary as *drift* observability. **Security-at-large:** ToolExecutor covers tool authz; add a plugin-trust policy (signature/permissions) for `plugin-runtime`, a credential-vault provider (Principal owns identity, vault owns secrets), and subprocess sandboxing in `agent-runtime` (OD's `sandbox-mode` is the seed). None of these are kernel changes — they're providers/policies.

## 6. Improved whole-system picture + biggest non-kernel risk

**The coherent one-line story:** Jini is a *headless, embeddable, agent-drivable daemon* (runs coding agents; exposes tools+providers; streams protocol events; over CLI/HTTP/MCP/sidecar). Everything else — chat-react, desktop-host, the automation control-plane — is an *optional satellite* that consumes that daemon. If that sentence stays true, the architecture is coherent.

Non-kernel changes to the locked plan: (a) protocol is the single event contract; chat-react consumes it (Seam B); (b) decouple the frontend track and sequence it *after* a shippable headless engine (§2); (c) lockstep `@jini/*` versioning (§5); (d) vocabulary firewall + agent-runtime-as-shared-spawn-primitive (Seam A); (e) minimal-runner-first, not cathedral (Seam D); (f) React packaging contract (Seam C).

**The single biggest non-kernel risk: betting months of agent-driven extraction on an unbuilt, unproven control plane.** The automation layer is the least-built part (project-runner doesn't exist as a runtime; ADS-memory is aspirational) yet it's what's supposed to *execute* the whole extraction. If it's flaky or over-engineered, the extraction stalls or drifts silently. De-risk: prove the extraction loop *manually* on the first 2–3 packages (protocol, platform, sidecar) with just the CI gates, *then* automate — and never let the control-plane's ambition outrun what the extraction actually needs. The kernel being right won't save you if the machine meant to build the other 40 pieces was itself the riskiest bet.
